# AI 开发规范与路线图

本文定义开发者和 AI 编程助手共同遵守的架构边界、变更流程与阶段目标。详细模块位置见 [[PROJECT_STRUCTURE]]，设计背景见 [[01-架构与数据流]]。

## 不可破坏的架构边界

- Domain 必须保持纯 Java，不依赖 Spring、HTTP、MyBatis、LangChain4j 或操作系统接口。
- `Score` 是生成、修改、校验和导出的唯一事实来源，不能在 DTO 或导出器中维护第二套音乐状态。
- Agent 负责理解意图和选择工具，不能直接写数据库、拼装最终 MusicXML 或绕过领域校验。
- Application 编排用例并依赖端口；Infrastructure 实现端口，不能把基础设施类型泄漏到业务核心。
- Controller、MCP 工具和 Agent 工具复用同一组 Application 用例，不复制业务规则。
- 所有生成和局部重写结果必须先通过完整 `Score` 校验，再创建新版本和导出物。

## 变更流程

1. 先确认需求影响的层级和事实来源。
2. 先修改领域模型或 Application 用例，再接入 API、MCP 或界面。
3. 为新增规则补充自动化测试，覆盖成功、边界和失败场景。
4. 运行受影响模块测试；涉及导出时重读 MIDI 并检查 MusicXML 结构。
5. 涉及 Guitar Pro 行为时完成人工导入验收。
6. 同步更新 [[PROJECT_STRUCTURE]] 及相关主题文档。

## 代码与数据要求

- 时间值使用精确分数模型，不以浮点数累计小节时值。
- 随机生成必须接受稳定种子，保证相同输入可复现。
- 局部重写只能替换指定闭区间小节，区间外内容必须保持不变。
- 数据库保存项目、任务、版本和文件元数据；乐谱聚合以版本化 JSON 快照保存。
- 错误响应必须可诊断，不能吞掉生成、校验、导出或持久化异常。
- 密钥只从环境变量或未提交的 `.env` 读取，日志中不得输出凭据。

## 完成定义

一个变更只有同时满足以下条件才算完成：

- 用户路径可运行，失败时有明确反馈。
- 新增或改变的规则有测试保护。
- 生成结果继续满足时值和可演奏性不变量。
- 不引入硬编码密钥、个人绝对路径或未说明的外部依赖。
- 文档、配置示例和代码行为保持一致。

完整测试矩阵与命令见 [[03-测试与验收]]。

## 路线图

### 当前：巩固单轨 MVP

- 稳定自然语言约束解析和差异化生成。
- 完善版本、局部重写、回滚与导出闭环。
- 保持 REST、SSE、MCP 和前端行为一致。

### 下一阶段：结构化创作

- 引入 SongPlan 与 RiffPlan，明确段落、动机和变奏关系。
- 增加和声进行、技法事件和更丰富的节奏表达。
- 扩展多轨模型，同时保持 Score 聚合一致性。

### 后续：质量与生态

- 建立可量化的音乐生成评测集。
- 改进音频分析准确度和可观测性。
- 扩展 DAW、谱面软件和外部 AI 客户端集成。
