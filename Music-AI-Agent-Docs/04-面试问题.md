# Music AI Agent 项目面试问题与真实回答

导航：[[00-首页]] · [[01-架构与数据流]] · [[PROJECT_STRUCTURE]]

> [!important] 使用方式
> 回答时先讲结论，再讲代码落点，最后主动说明限制。不要把当前项目描述成“自研音乐大模型”或“完整多轨生成平台”；当前核心是 **LangChain4j 结构化理解 + Java 约束生成 + 领域校验 + MIDI/MusicXML 导出**。

## 一分钟项目介绍

### Q1：请介绍一下这个项目

这是一个面向吉他手和音乐创作者的 Music AI Agent。用户输入自然语言创作要求，例如“生成 8 小节、120 BPM、E 小调、标准调弦的摇滚 Riff”，系统将其解析为结构化 `CreationConstraints`，再由纯 Java 生成器创建可演奏的音乐事件，通过领域规则校验小节时值和吉他弦品，最后保存不可变版本并导出 Guitar Pro 8 可导入的 MIDI 与 MusicXML。

主链路是：

```text
HTTP/Agent 请求
→ MusicProjectService 创建持久化任务
→ RequirementParser 解析 CreationConstraints
→ GuitarRiffGenerator 生成 Score
→ Score.validate() 确定性校验
→ ProjectStore 保存 JSON 版本快照
→ MidiExporter / MusicXmlExporter
→ SSE 通知前端
```

技术栈包括 Spring Boot 3、Java 21、LangChain4j、可配置OpenAI兼容模型、MyBatis-Plus、MySQL/H2、Vue 3、MCP、Docker Compose、JUnit 5 和 Testcontainers。

### Q2：这个项目最核心的技术难点是什么？

难点不是调用一次大模型，而是把不稳定的自然语言输出转换成可验证、可修改、可导出的确定性音乐结构：

1. 大模型只提取约束，不能直接生成最终 MusicXML。
2. `RhythmicDuration` 使用分数而不是 `double`，保证小节时值精确。
3. `GuitarTuning` 将 MIDI 音高映射到具体弦和品位，过滤不可演奏音符。
4. `Score` 是生成、重写、校验和导出的唯一事实来源。
5. 每次重写创建新版本，不静默覆盖原乐谱。

当前真正体现项目差异化的代码是 `GuitarRiffGenerator`、`domain` 包和两个 Exporter，而不是普通 Controller 或 Mapper。

## 架构与分层

### Q3：为什么没有使用传统的 Controller → Service → ServiceImpl → Mapper？

项目采用模块化单体和端口适配器思想：

| 层 | 真实职责 |
|---|---|
| API | HTTP、DTO、校验、状态码、SSE |
| Application | 用例编排、任务状态、版本和导出流程 |
| Domain | 纯 Java 音乐模型与不变量 |
| Agent | LangChain4j AI Service、Memory、Tools |
| Infrastructure | MyBatis、模型配置、安全、系统集成 |
| Export | MIDI、MusicXML 格式转换 |

`ProjectStore` 是应用层端口，`MyBatisProjectStore` 是基础设施实现。这样 `MusicProjectService` 不依赖 Mapper，领域对象也不依赖 Spring 或数据库实体。`ServiceImpl` 只有在确实存在多个业务实现时才有价值，为接口机械增加 Impl 会增加跳转成本，却没有形成真正边界。

### Q4：为什么当前不拆微服务？

当前首要风险是音乐中间表示、可演奏性和 Guitar Pro 导出兼容性，不是独立扩缩容。模块化单体可以在同一进程内保持事务和调试简单，避免过早引入服务发现、分布式事务、链路追踪和跨服务数据一致性。

物理目录已经分为后端、前端和可选 Python 服务；Java 内部先通过包和端口维持边界。只有某个能力出现独立部署、资源隔离或扩缩容需求时，才考虑拆分。

### Q5：为什么领域层必须保持纯 Java？

音乐规则需要高频单元测试，而且不应因为 Web、ORM 或模型框架变化而重写。`Measure`、`RhythmicDuration`、`GuitarTuning` 和 `Score` 不依赖 Spring、LangChain4j、MyBatis 或 HTTP，因此可以直接构造测试，也可以被 REST、MCP、CLI 或未来桌面端复用。

## 请求、异步任务与 SSE

### Q6：一次生成请求是怎样执行的？

`ProjectController.generate()` 接收请求后调用 `MusicProjectService.generate()`。Service 先通过 `requireProject()` 确认项目存在，再用 `store.createTask()` 保存 `PENDING` 任务和用户消息，然后通过 `TaskExecutor` 执行 `runGeneration()`，最后立即返回任务 ID。

后台流程依次进入：

```text
PARSING_REQUIREMENTS
→ GENERATING
→ VALIDATING
→ EXPORTING
→ COMPLETED / FAILED
```

前端得到 HTTP 202 后，可以轮询 `/api/tasks/{taskId}`，也可以通过项目 SSE 连接接收阶段事件。

### Q7：为什么必须先持久化任务，再异步执行？

如果只在线程池中运行任务，应用重启或请求结束后就缺少可查询状态。先保存任务有三个作用：

1. 客户端立即得到稳定 `taskId`。
2. 所有状态变化和错误都能写回数据库。
3. 应用启动时 `InterruptedTaskRecovery` 可以把遗留的非终态任务标记为失败，避免永久显示 `RUNNING`。

当前线程池仍是单机实现，不是可靠消息队列；它解决了请求线程阻塞和任务可追踪问题，但不保证宕机后自动续跑。

### Q8：SSE 是怎样实现并发安全的？

`ProjectEventBroker` 按项目 ID 保存 `SseEmitter` 列表。外层映射需要线程安全，项目内连接使用 `CopyOnWriteArrayList`，适合“广播读取多、连接增删少”的场景。写操作加锁并复制数组，遍历读取稳定快照，因此连接超时或断开时可以安全移除。

当前 SSE 是进程内实时通知，没有事件持久化和历史重放保证。客户端断线后应通过任务查询接口重新获得最终状态。

### Q9：当前任务事件设计有什么可改进点？

当前 `transition()` 同时更新任务状态并发布一个携带状态字符串的事件，而 `EXPORT_COMPLETED` 需要携带导出物列表，所以完成阶段没有直接复用 `transition()`。

更清晰的设计是拆成：

```text
TASK_STATUS_CHANGED → 只表示状态变化
VALIDATION_COMPLETED → 携带校验结果
EXPORT_COMPLETED → 携带 MIDI/MusicXML 导出物
```

另外，当前部分事件名与执行时机仍可进一步严格对齐，例如应该在真正校验完成后再发布 `VALIDATION_COMPLETED`。

## LangChain4j 与可配置模型

### Q10：项目中到底是谁触发了 LangChain4j？

分为创建和调用两个阶段：

1. Spring 启用 `llm` Profile 时执行 `LlmConfiguration.musicCreatorAgent()`。
2. `AiServices.builder(MusicCreatorAgent.class)` 创建动态代理，并注入 `ChatModel`、`MusicCreationTools` 和项目级 Chat Memory。
3. HTTP 请求进入 `ProjectController.chat()`。
4. `agent.chat(projectId, prompt)` 才真正触发 LangChain4j 读取注解、组装消息、调用当前配置模型并处理工具调用。

因此，Spring MVC 唤醒 Controller，`agent.chat()` 唤醒 LangChain4j 动态代理，具体`ChatModel`最终向配置的模型端点发送HTTP请求。

### Q11：`MusicCreatorAgent` 没有实现类，为什么能工作？

它是 LangChain4j AI Service 接口。`AiServices.builder(...).build()` 在运行时创建代理实现。代理读取：

- `@SystemMessage`：加载系统提示词。
- `@UserMessage` 和 `@V`：填充项目 ID 与用户消息。
- `@MemoryId`：按项目隔离最近 20 条消息。
- 注册的 Tools：处理模型产生的工具调用。

接口返回 `String`，因此代理将模型最终文本作为 `chat()` 返回值。Controller 再用 `AgentChatResponse` 包装成 JSON。

### Q12：为什么 Controller 使用 `ObjectProvider<MusicCreatorAgent>`？

`MusicCreatorAgent`只在`llm` Profile下创建。如果直接构造器注入，未配置模型时整个Spring应用会启动失败。`ObjectProvider.getIfAvailable()`把Agent变成可选能力：离线规则生成仍能运行，只有调用`/chat`时才检查Agent是否存在。

### Q13：`MusicCreatorAgent` 和 `CreationConstraintsAiService` 有什么区别？

`MusicCreatorAgent` 负责对话、记忆和工具编排，返回面向用户的文本；`CreationConstraintsAiService` 只负责把一句自然语言解析为结构化 `CreationConstraintsResponse`。

后者由`AiRequirementParser`调用，并在模型输出后继续执行本地默认值、枚举转换和值对象校验。模型输出不能绕过Java校验直接进入生成器。

### Q14：没有模型Key，为什么项目仍能生成音乐？

默认Profile使用`RuleBasedRequirementParser`，通过中英文关键词和默认值解析约束；真正音符生成由Java `GuitarRiffGenerator`完成。因此无Key时可以完成规则生成、校验和导出，但不能使用模型对话和更灵活的自然语言解析。

启用`llm` Profile时需要通过`LLM_PROVIDER`、`LLM_BASE_URL`、`LLM_MODEL`和`LLM_API_KEY`配置模型，代码和日志均不保存明文密钥。

### Q15：如何避免大模型幻觉破坏乐谱？

项目没有让模型直接输出数据库 ID、文件路径或最终 MusicXML。模型只输出受限 DTO，之后依次经过：

```text
对象反序列化
→ 默认值与枚举转换
→ CreationConstraints 构造校验
→ GuitarRiffGenerator
→ Score.validate()
→ Exporter
```

即使模型返回错误调弦、拍号或越界复杂度，也会在进入领域模型前失败，而不是生成不可信文件。

## 音乐生成与领域模型

### Q16：项目生成的是 AI 音乐，还是规则音乐？

当前是“AI理解 + 规则生成”。配置的聊天模型理解用户意图并提取结构化约束；音符、休止、节奏和弦品由确定性Java生成器产生。它不是自训练音乐基础模型，也没有声称可以生成任意多轨完整作品。

这样设计可以先验证从自然语言到 Guitar Pro 文件的纵向链路，并保证生成结果可复现、可校验。后续可引入结构化 `SongPlan`、`RiffPlan` 或 Python 模型，但仍必须经过统一领域表示和确定性校验。

### Q17：如何避免不同提示词总是生成相同片段？

早期固定音高数组已经被替换。当前生成器综合使用：

- 调性与大小调模式选择音阶。
- rock/metal/blues 风格选择五声音阶或布鲁斯音阶。
- `RhythmicFeel` 决定每拍细分。
- `MusicalMood` 影响休止概率和音区。
- `complexity` 影响可用音级和变奏概率。
- 原始提示词生成稳定 `variationSeed`。

`mixSeed()` 将种子、调性、风格、情绪、节奏和复杂度混合。相同输入可复现，不同提示词或关键约束应产生不同事件序列，并有自动化测试保证。

### Q18：为什么时值不用 `double`？

浮点数不能精确表示很多分数，连续相加可能出现 `0.999999...`，导致合法小节校验失败。`RhythmicDuration` 保存分子和分母，构造时约分，并提供精确加法、比较和 PPQ tick 转换。

例如 4/4 小节的期望时值是 `1/1` 个全音符，四个四分音符精确相加后必须等于 `1/1`。这是领域不变量，不依赖容差判断。

### Q19：如何保证音符可以在吉他上演奏？

`GuitarTuning` 保存六根空弦的 MIDI 音高。对于目标 `Pitch`，每根弦的品位等于：

```text
目标 MIDI 音高 - 空弦 MIDI 音高
```

只有品位位于 `0..maximumFret` 才是候选位置。候选结果优先低品位，同品位优先较小弦号，以保证选择稳定。生成器通过 `requirePlayablePosition()` 要求每个音符都有合法弦品，否则立即失败。

当前只实现基础单音/和弦合法性，还没有完整的跨事件左手移动成本和高级指法优化。

### Q20：`Score.validate()` 主要验证什么？

当前验证至少包括：

1. 每个小节事件总时值与拍号一致。
2. 轨道、小节和事件结构非空且编号合法。
3. 音高在 MIDI 合法范围。
4. 音符的弦品位置与当前调弦对应。
5. 和弦不能在同一时刻重复使用同一根弦。

Exporter 在写文件前再次调用 `score.validate()`，避免未验证领域对象直接落盘。

### Q21：局部重写怎样保证不影响其他小节？

`runRewrite()` 读取当前 `Score`，生成一份与原工程小节数、拍号和调弦兼容的候选乐谱，然后 `replaceMeasures()` 复制原小节列表，只替换 `[fromMeasure, toMeasure]` 闭区间。合并后的完整 `Score` 再次校验并保存为新版本。

自动化测试会比较目标范围之外的小节是否保持相等。

需要诚实说明：当前重写约束仍采用解析结果中的 tempo 和 key，可能改变新版本的全局速度或调性。若产品要求严格局部修改，应保留 `original.tempo()` 和 `original.keySignature()`，把全局设置修改拆成独立用例。

## 持久化与版本

### Q22：为什么乐谱保存为 JSON 快照，而不是每个音符一行？

MVP 的主要操作是整版读取、生成新版本、局部替换后整体校验和导出。JSON 快照可以保持 `Score` 聚合的一致性，也方便版本回滚；项目、任务、消息和导出物元数据仍然使用关系表。

如果把每个音符拆成行，会引入大量表、排序字段和跨表事务，却不符合当前查询模式。未来需要跨项目音符统计或在线协作时，可以增加派生索引或事件存储，而不是现在过度规范化。

### Q23：`currentVersion` 为什么称为版本指针？

`music_project.current_version` 只表示当前激活的版本号，完整乐谱在 `project_version.score_json` 中。回滚时仅移动 `currentVersion`，不会删除后续版本，所以历史记录仍然可审计和再次切换。

`currentScore()` 根据项目取得当前版本号，再读取对应 `StoredVersion.scoreJson()`，通过 Jackson 反序列化为领域 `Score`。

### Q24：保存版本时可能有什么并发问题？

当前 `MyBatisProjectStore.saveVersion()` 使用 `synchronized` 在单 JVM 内串行执行：读取 `currentVersion + 1`、插入版本、更新项目指针。但多实例部署时，每个 JVM 有自己的锁，仍可能同时分配相同版本号。

生产级方案应结合数据库唯一约束，并使用行锁、乐观锁版本字段或数据库序列完成跨实例并发控制。当前实现适合单实例 MVP，但不应被描述为分布式安全。

### Q25：为什么同时支持 H2 和 MySQL？

默认本地开发和普通测试使用 H2，减少启动依赖；Docker Compose 和 `mysql` Profile 使用 MySQL 8。两套配置共用核心表结构概念，MySQL 持久化链路由 Testcontainers 集成测试验证。

H2 不是生产数据库。涉及 SQL 方言、索引、锁和事务行为的功能必须在 MySQL 测试中再次验证。

## 导出与文件安全

### Q26：MIDI 与 MusicXML 分别怎样生成？

`MidiExporter` 使用 Java MIDI API，采用 480 PPQ，将领域分数时值转换为 tick，并写入速度、拍号、Note On/Off 事件。休止符不写 MIDI 消息，但会推进时间轴；和弦音共享相同起止时间。

`MusicXmlExporter` 使用 DOM 构建 MusicXML 4.0，写入调号、拍号、速度、吉他谱弦号和品位。和弦通过后续 `<note>` 的 `<chord/>` 表达，而不是自定义结构。

两者都以经过校验的 `Score` 为输入，不依赖模型直接拼接文本。

### Q27：导出路径为什么要 `normalize()` 和 `startsWith()`？

目标目录按 `exportRoot/projectId/v{version}` 构建。`normalize()` 消除 `.` 和 `..`，然后 `startsWith(exportDirectory)` 确认最终路径仍位于允许的导出根目录，防止路径穿越。

当前项目 ID 是服务端 UUID，本身较安全，但保留检查可以防止未来 ID 来源变化、数据导入或数据库污染后把文件写到根目录之外。

### Q28：Guitar Pro 8 是怎样集成的？

项目不假设 Guitar Pro 存在公开第三方 API。核心链路是导出标准 MIDI 和 MusicXML；Windows 服务端可通过 `GuitarProConnector` 调用系统文件关联打开导出物。UI Automation 不属于核心数据链路，也不应该用于保证生成正确性。

## MCP 与安全

### Q29：MCP 在项目中起什么作用？

MCP 将已有 Application 用例包装成标准工具，使支持 MCP 的外部 AI 客户端可以创建项目、启动生成、查询任务、验证、列出导出物、重写和回滚。

`MusicMcpToolCatalog` 声明输入/输出 Schema、行为提示和 Bean Validation，真正执行仍委托 `MusicProjectService`。因此 MCP 是协议适配层，不是另一套业务逻辑，也不是音乐模型。

其中`generate_guitar_riff_from_constraints`专门接收外部模型已经解析好的结构化约束，不调用服务器`RequirementParser`。因此开发者可以在自己的MCP客户端使用自己的模型和Key，服务器只承担确定性音乐业务。

生成和重写工具虽然通过同步 MCP 调用返回，但返回的是持久化异步任务，客户端仍需轮询状态。

### Q30：项目已经有内部 Agent，为什么还需要 MCP？内部 Agent 会调用 MCP 吗？

内部 `MusicCreatorAgent` 服务于本项目 `/chat` 接口，由服务器配置的模型选择并调用 Java `MusicCreationTools`；它不会绕一圈通过 HTTP 调用自己的 MCP Server。生成工具直接提交结构化约束到 `MusicProjectService`，因此一次对话生成只需模型完成一次理解和工具选择。

MCP 面向进程外的 AI 客户端或开发者自建 Agent。它把同一组应用用例暴露成标准工具，使外部模型能够使用自己的 Key 完成编排。两条路径共享 Application 和 Domain，但入口、模型所有者与认证方式不同。

### Q31：如何保护访问密钥和 MCP Token？

模型 Key、项目访问密钥和 MCP Token 都通过环境变量或未提交配置注入，不进入源码和日志。REST 支持 Header 密钥或 HttpOnly、SameSite=Strict 的 HMAC 会话 Cookie；比较敏感值时使用常量时间比较。MCP 端点使用 Bearer Token 过滤器。

Cookie 中保存的是有时效的签名 Token，不是原始访问密钥。生产 HTTPS 环境应启用 Secure Cookie。

### Q32：为什么 `/chat` 开始时要 `requireProject(projectId)`？

`projectId` 同时是 LangChain4j 的 `@MemoryId`。如果不先检查，任意不存在的 ID 都可以创建孤立聊天记忆并触发付费模型请求。提前检查可以立即返回 404，避免无效调用；即使工具层内部还有二次检查，Controller 的前置检查仍能保护聊天上下文和成本。

## 测试、缺陷与演进

### Q33：项目有哪些关键测试？

测试重点按风险分层：

- `RhythmicDurationTest`、`MeasureTest`：精确时值和小节不变量。
- `GuitarTuningTest`、`GuitarPlayabilityTest`：标准/Drop D 弦品与可演奏性。
- `GuitarRiffGeneratorTest`：差异性、复现性、调性和拍号。
- `MeasureRewriteTest`：非目标小节保持不变。
- `ExportersTest`：MIDI 可重新读取、MusicXML 结构与调号。
- `MusicProjectServiceIntegrationTest`：生成、版本和导出纵向链路。
- `MusicMcpIntegrationTest`：协议、认证和工具调用。
- `MySqlProjectStoreTest`：Testcontainers MySQL 8 持久化。

真实LLM测试默认跳过，普通测试不能依赖网络和真实API Key。

### Q34：当前项目有哪些真实限制？

必须主动说明：

1. 当前只生成单轨吉他，不是通用多轨音乐模型。
2. `SongPlan` 仍是初步结构摘要，没有完整和声进行与段落功能模型。
3. 技法枚举尚未完整进入音乐事件和导出。
4. 局部重写会生成完整候选乐谱，效率还可优化。
5. 任务执行依赖单机线程池，宕机后只标记失败，不自动续跑。
6. SSE 没有持久化与事件重放。
7. 单 JVM 的 `synchronized` 不能解决多实例版本并发。
8. Python BPM 是幅度包络自相关初估，不是专业节拍转录。

主动说明边界通常比夸大能力更能体现工程判断。

### Q35：如果继续开发，你会优先做什么？

优先级应沿着当前纵向链路演进：

1. 引入结构化 `SongPlan` 和 `RiffPlan`，表达和声、段落、轮廓与技法。
2. 修正状态事件语义，并为 SSE 增加可恢复的任务状态读取。
3. 将局部重写限制为不改变全局 tempo/key，或增加显式全局修改用例。
4. 完善指法跨度、跨事件移动成本和自定义调弦。
5. 增加乐谱预览、MIDI 播放和项目列表完整体验。
6. 需要多实例时，再引入可靠队列和数据库级并发控制。
